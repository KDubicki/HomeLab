## Overview
A single **Dell OptiPlex 7070** Proxmox node being built, in code, as a cloud-analog DevOps + Data-Engineering platform (see `change-plan/0001`). Today the platform is at its **foundation stage**: the host is prepared and storage/identity are in place, but no workloads run yet. Honest framing: single node → this demonstrates cloud-native *patterns*, not production HA.

```
LAN 192.168.0.0/24  (gw 192.168.0.1)
        │
        └── Proxmox host  "pve"  192.168.0.113  ── bridge vmbr0 ── nic0 (1 GbE)
              ├── local-lvm   (NVMe, 816 GiB)   VM/CT root disks
              ├── ssd-data    (SATA SSD, 233 GiB) stateful data volumes  [empty]
              └── guests: none yet
```
No private subnet / edge yet (planned in `change-plan/0004`); everything is on `vmbr0`.

Storage tiering is deliberate: OS/VM disks on fast NVMe, stateful data (future Vault/Postgres/MinIO) isolated on the SSD pool (`change-log/0002`).

## Identities & access
| Identity | Realm | Role / scope | Purpose |
| :--- | :--- | :--- | :--- |
| `root@pam` | pam | superuser | host administration (SSH) |
| `terraform@pve` | pve | `TerraformProv` (21 privileges) at `/`, propagate | least-privilege IaC identity for Terraform (0003) |

- **API token**: `terraform@pve!provider` (privsep 0) — the credential Terraform will authenticate with.
- **Custom role** `TerraformProv`: 21 PVE-9-valid privileges (Datastore.\*, Pool.Allocate, Sys.\*, VM.\* lifecycle/config, SDN.Use). Deliberately **no** `VM.Monitor` (absent in PVE 9).
- **Secrets**: the token currently lives in a gitignored `terraform/secrets.auto.tfvars`; it will migrate to **Vault** (`change-plan/0005`). No Vault yet. Secret values are never stored in this repo or this doc.
