This is a **reference of reality** (what exists now); for *why*, see `change-plan/`/`change-log/`; for *how* it was built, see `runbooks/` (deleted once Executed) and the repo's `terraform/`/`ansible/` trees.

## Overview
A single **Dell OptiPlex 7070** Proxmox node being built, in code, as a cloud-analog DevOps + Data-Engineering platform (see `change-log/0001`). The platform has a working **IaC control plane** (`change-log/0003`), a **private-subnet network** (`change-log/0004`), and now a **secrets manager** built on the private subnet: a single-node HashiCorp Vault (TLS, Raft storage, KV v2 + database engines mounted, an AppRole wired into Terraform). Two guests run persistently (`edge`, `vault`); the golden template remains stopped. Honest framing: single node → this demonstrates cloud-native *patterns* (VPC + bastion, centralized secrets with least-privilege auth), not production HA, physical network isolation, or a highly-available Vault cluster.

```
LAN 192.168.0.0/24  (gw 192.168.0.1)
        │
        └── Proxmox host  "pve"  192.168.0.113  ── bridge vmbr0 ── nic0 (1 GbE)
              │
              ├── local-lvm   (NVMe, 816 GiB, ~0.44% used)   VM/CT root disks
              │     ├── VMID 9000 "debian13-cloud"  — golden VM template, stopped
              │     ├── VMID  101 "edge"             — LXC, running (4 GiB disk)
              │     └── VMID  102 "vault"            — LXC, running (8 GiB disk)
              ├── ssd-data    (SATA SSD, 233.5 GiB, ~0.06% used)  stateful data volumes
              │     └── vm-102-disk-0 (5 GiB)  — Vault's Raft data mount (/vault/data)
              │
              └── vmbr1  (10.10.10.0/24, no bridge-ports — no LAN uplink)
                    ├── edge (LXC 101)   eth1 10.10.10.1   — NAT gateway + SSH bastion + DHCP/DNS
                    └── vault (LXC 102)  eth0 10.10.10.10  — HashiCorp Vault, TLS :8200, private-subnet only (no LAN NIC)
                          (k3s/pg/minio attach here in 0006–0007; none yet)
```
`edge` is dual-homed: `eth0` → `vmbr0` (`192.168.0.10`, LAN-reachable), `eth1` → `vmbr1` (`10.10.10.1`, private gateway). `vault` has a single NIC on `vmbr1` only — it is reachable **only** by SSH-jumping through `edge`, or via an SSH local-port-forward tunnel for its HTTPS API (`ssh -L 8200:10.10.10.10:8200 deploy@192.168.0.10`). Both patterns were proven end-to-end during 0004 (throwaway DHCP client `priv-test`, VMID 998, destroyed after verification) and 0005 (Vault init/unseal, secrets-engine setup, and a live `terraform plan` reading a secret from Vault over the tunnel).

Storage tiering is deliberate: OS/VM/CT disks on fast NVMe, stateful data isolated on the SSD pool (`change-log/0002`). `vault` is the first guest to use a second, separate mount point (`mp0`) alongside its rootfs — the `terraform/modules/lxc` module gained an optional `data_disk` block in 0005 to support this.

## IaC control plane (`change-log/0003`, extended by 0004 and 0005)
| Component | Detail |
| :--- | :--- |
| Provisioning | Terraform ≥1.9, provider `bpg/proxmox` (installed: v0.111.1) + `hashicorp/vault` (installed: v5.10.1, added in 0005), state local & gitignored (`terraform/*.tfstate`) |
| VM path | `terraform/modules/vm` clones golden template 9000; two-phase guest-agent flip (Ansible installs it, second apply enables the channel) |
| LXC path | `terraform/modules/lxc` — builds an unprivileged container from a downloaded `vztmpl`; parametrized multi-NIC `network_interfaces`; an optional `data_disk` block (added in 0005) adds a second `mount_point` volume on a separate datastore (used by `vault`'s Raft storage) |
| Container template | Debian 13 `debian-13-standard_13.1-2_amd64.tar.zst`, downloaded via `proxmox_virtual_environment_download_file` (content type `vztmpl`, md5-checksummed) from `download.proxmox.com` |
| Network as code | `proxmox_network_linux_bridge.vmbr1` — the private bridge is Terraform-managed, not GUI-created |
| Secrets as code | `terraform/secrets.tf` — a `vault` provider (AppRole login, `skip_child_token = true` to stay within the least-privilege policy) plus a `vault_kv_secret_v2` data source feed the `proxmox` provider's API token (`change-log/0005`) |
| Configuration | Ansible ≥2.16; `baseline` role guards `qemu-guest-agent` with `ansible_virtualization_type != 'lxc'`; `edge` role: IPv4 forwarding, nftables ruleset, dnsmasq (now with a static host record for `vault.lab.internal`); new `vault` role (0005): installs Vault 1.17.6, generates a self-signed TLS cert, writes the Raft/TLS config and systemd unit |
| LXC bootstrap pattern | Proxmox writes the SSH key straight into the container's `root` account at creation (no cloud-init) → Ansible connects as `root` once, creates `deploy`, disables root/password login. **Operational note from 0005**: this means `root` SSH access is a *one-time* window — every later connection (Ansible re-runs, manual admin) must use `deploy` + `sudo`, and this OpenSSH client build (10.2p1) does not propagate `-i` across a `ProxyJump` hop with no agent key loaded, so jumps through `edge` use an explicit `ProxyCommand` instead |

## Secrets management (`change-log/0005`)
| Component | Detail |
| :--- | :--- |
| Service | HashiCorp Vault 1.17.6 OSS, systemd-managed, TLS listener on `0.0.0.0:8200` (self-signed cert, SAN `vault.lab.internal` + `10.10.10.10` + `127.0.0.1`) |
| Storage | Integrated Raft, single node, data at `/vault/data` (the `ssd-data` mount) — `disable_mlock = true` (an unprivileged LXC cannot grant `CAP_IPC_LOCK`) |
| Seal | Shamir, 5 shares / 3 threshold. Initialized and unsealed. Unseal keys held offline only (never in this repo) |
| Secrets engines | `kv/` (KV v2) — holds `kv/proxmox/terraform` (the Proxmox API token); `database/` mounted but not yet connected (deferred to `change-plan/0007`, once Postgres exists) |
| Auth | AppRole `terraform`, bound to a policy scoped to `kv/data/proxmox/*` read only, `token_ttl=30m` / `token_max_ttl=2h` / `secret_id_ttl=2160h` (90 days) |
| Access pattern | No direct LAN route — reached via `ssh -J`/`ProxyCommand` through `edge` for shell access, or an SSH local-port-forward tunnel (`-L 8200:10.10.10.10:8200`) for the HTTPS API, which Terraform uses on every `plan`/`apply` |
| Root token | Root tokens are not kept standing — generated on demand via `vault operator generate-root` using 3 of the 5 unseal keys, and revoked immediately after use |

## Identities & access
| Identity | Realm | Role / scope | Purpose |
| :--- | :--- | :--- | :--- |
| `root@pam` | pam | superuser | host administration (SSH) |
| `terraform@pve` | pve | `TerraformProv` (22 privileges) at `/`, propagate | least-privilege IaC identity for Terraform — unchanged in privilege scope by 0004/0005 (no new PVE privileges were needed; the credential itself now lives in Vault, not a gitignored file) |

- **API token**: `terraform@pve!provider` — the credential Terraform authenticates with, **now sourced from Vault** (`kv/proxmox/terraform`) instead of a gitignored tfvars file.
- **In-guest (VMs)**: bootstrap user `debian` (cloud-init, key-only) → Ansible creates `deploy`.
- **In-guest (LXCs)**: bootstrap as `root` (Proxmox-injected key) → Ansible creates `deploy`, then disables root/password SSH. This is a one-shot window — see the IaC table note above.
- **Bastion access**: private-subnet guests are reached only via a jump through `edge` (`deploy@192.168.0.10`) — no direct LAN route exists to `10.10.10.0/24`.
- **Secrets**: the Proxmox token and Terraform's Vault AppRole credentials are the only local secrets remaining, both gitignored (`terraform/secrets.auto.tfvars`); the Proxmox token's authoritative copy is now Vault's `kv/proxmox/terraform`. Secret values are never stored in this repo or this doc.
