## Overview
A single **Dell OptiPlex 7070** Proxmox node being built, in code, as a cloud-analog DevOps + Data-Engineering platform (see `change-log/0001`). The platform now has a working **IaC control plane**: a golden VM template and a reusable Terraform module + Ansible role, proven end-to-end. No guest runs persistently yet — the next slice (`change-plan/0004`) builds the private-subnet network. Honest framing: single node → this demonstrates cloud-native *patterns*, not production HA.

```
LAN 192.168.0.0/24  (gw 192.168.0.1)
        │
        └── Proxmox host  "pve"  192.168.0.113  ── bridge vmbr0 ── nic0 (1 GbE)

              ├── ssd-data    (SATA SSD, 233 GiB, 0% used)  stateful data volumes  [empty]
```

No private subnet / edge yet (planned in `change-plan/0004`); everything is on `vmbr0`.

## IaC control plane (`change-log/0003`)
| Component | Detail |
| :--- | :--- |
| Provisioning | Terraform ≥1.9, provider `bpg/proxmox` (installed: v0.111.1), state local & gitignored (`terraform/*.tfstate`) |
| Golden template | VMID 9000, built from a stock Debian 13 (trixie) generic cloud image via `download_file` (content type `import` on `local`) — zero in-image baking |
| Clone module | `terraform/modules/vm` — clones template 9000, sets vCPU/RAM/disk/network/cloud-init per guest; `agent_enabled` var (starts `false`, flipped to `true` post-Ansible — see below) |
| Configuration | Ansible ≥2.16, role `ansible/roles/baseline`: qemu-guest-agent, `deploy` sudo/key-only user, SSH hardening (no root, no password auth), `node_exporter` on `:9100` |
| Guest-agent pattern | Two-phase: VM created with the agent channel disabled (avoids Terraform blocking pre-install) → Ansible installs `qemu-guest-agent` → a second `terraform apply` flips `agent.enabled = true`, hot-plugging the virtio channel with no guest reboot |
| Image source | Pinned to a direct mirror (`docs/platform-conventions.md`), not `cloud.debian.org` directly — its geo-redirect was measured at ~56s, exceeding Proxmox's own download timeout |

## Identities & access
| Identity | Realm | Role / scope | Purpose |
| :--- | :--- | :--- | :--- |
| `root@pam` | pam | superuser | host administration (SSH) |
| `terraform@pve` | pve | `TerraformProv` (22 privileges) at `/`, propagate | least-privilege IaC identity for Terraform |

- **API token**: `terraform@pve!provider` (privsep 0) — the credential Terraform authenticates with.
- **Custom role** `TerraformProv`: 22 PVE-9-valid privileges (`Datastore.*`, `Pool.Allocate`, `Sys.*`, `VM.*` lifecycle/config, `SDN.Use`, `VM.GuestAgent.Audit`). Deliberately **no** `VM.Monitor` (absent in PVE 9) and no `VM.GuestAgent.Unrestricted` (read-only agent audit is enough; no exec-via-agent).
- **In-guest**: bootstrap user `debian` (cloud-init, key-only) on first boot; Ansible then creates `deploy` (sudo, key-only) and disables root/password SSH login.
- **Secrets**: the Proxmox token lives in a gitignored `terraform/secrets.auto.tfvars`; it will migrate to **Vault** (`change-plan/0005`). No Vault yet. Secret values are never stored in this repo or this doc.
