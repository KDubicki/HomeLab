# System Documentation (As-Built)

> **As-of:** 2026-07-09 · **Proxmox VE 9.2.4** (kernel 7.0.14-2-pve) · generated by the `architect-docs` skill from a live `ssh proxmox` scan. This is a **reference of reality** (what exists now); for *why*, see `change-plan/`/`change-log/`; for *how* it was built, see `runbooks/` (deleted once Executed) and the repo's `terraform/`/`ansible/` trees.

## Overview
A single **Dell OptiPlex 7070** Proxmox node being built, in code, as a cloud-analog DevOps + Data-Engineering platform (see `change-log/0001`). The platform has a working **IaC control plane** (`change-log/0003`), a **private-subnet network** (`change-log/0004`), a **secrets manager** on the private subnet (single-node HashiCorp Vault — `change-log/0005`), a **compute tier**: single-node k3s with a live HashiCorp Vault Agent Injector (`change-log/0006`), and now a **managed-services tier**: PostgreSQL 17 and MinIO, each with Vault-mediated credentials (`change-log/0007`). Five guests run persistently (`edge`, `vault`, `k3s-1`, `pg`, `minio`); the golden template remains stopped. Honest framing: single node → this demonstrates cloud-native *patterns* (VPC + bastion, centralized secrets with least-privilege auth, managed-Kubernetes-style compute, RDS/S3-style managed services), not production HA, physical network isolation, or highly-available Vault/k8s/Postgres/MinIO.

```
LAN 192.168.0.0/24  (gw 192.168.0.1)
        │
        └── Proxmox host  "pve"  192.168.0.113  ── bridge vmbr0 ── nic0 (1 GbE)
              │
              ├── local-lvm   (NVMe, 816 GiB, ~1.4% used)   VM/CT root disks
              │     ├── VMID 9000 "debian13-cloud"  — golden VM template, stopped
              │     ├── VMID  101 "edge"             — LXC, running (4 GiB disk)
              │     ├── VMID  102 "vault"            — LXC, running (8 GiB disk)
              │     ├── VMID  110 "k3s-1"            — VM, running (60 GiB disk)
              │     ├── VMID  120 "pg"               — LXC, running (8 GiB disk)
              │     └── VMID  121 "minio"            — LXC, running (8 GiB disk)
              ├── ssd-data    (SATA SSD, 233.5 GiB, ~1.5% used)  stateful data volumes
              │     ├── vm-102-disk-0 (5 GiB)    — Vault's Raft data mount (/vault/data)
              │     ├── vm-120-disk-0 (40 GiB)   — Postgres data mount (/pg/data)
              │     └── vm-121-disk-0 (120 GiB)  — MinIO data mount (/minio/data)
              │
              └── vmbr1  (10.10.10.0/24, no bridge-ports — no LAN uplink)
                    ├── edge (LXC 101)   eth1 10.10.10.1   — NAT gateway + SSH bastion + DHCP/DNS + DNAT (80/443 → k3s-1)
                    ├── vault (LXC 102)  eth0 10.10.10.10  — HashiCorp Vault, TLS :8200, private-subnet only (no LAN NIC)
                    ├── k3s-1 (VM 110)   eth0 10.10.10.20  — single-node k3s v1.36.2+k3s1, private-subnet only
                    ├── pg (LXC 120)     eth0 10.10.10.30  — PostgreSQL 17, private-subnet only
                    └── minio (LXC 121)  eth0 10.10.10.40  — MinIO object store, private-subnet only
```
`edge` is dual-homed: `eth0` → `vmbr0` (`192.168.0.10`, LAN-reachable), `eth1` → `vmbr1` (`10.10.10.1`, private gateway). `vault`, `k3s-1`, `pg`, and `minio` each have a single NIC on `vmbr1` only — reachable **only** by SSH-jumping through `edge`, or via an SSH local-port-forward tunnel for an API (Vault: `-L 8200:10.10.10.10:8200`; k3s kube-API: `-L 6443:10.10.10.20:6443`). `edge` additionally DNATs LAN-side `tcp/80,443` straight to `k3s-1` (`change-log/0006`) — the one deliberate exception to "bastion-only": ingress traffic is meant to be reached directly, the way a cloud ALB fronts a private-subnet target, while the cluster's admin plane (kube-API) and the managed-services tier stay tunnel/bastion-only like Vault's. These patterns were proven end-to-end during 0004 (throwaway DHCP client `priv-test`, destroyed after verification), 0005 (Vault init/unseal, secrets-engine setup, a live `terraform plan` reading a secret over the tunnel), 0006 (a sample Kubernetes Deployment/Ingress reachable via the DNAT, with a Vault-injected secret, and `kubectl` reachable via the tunnel), and 0007 (two k3s demo Jobs: a Vault Agent sidecar minting a live Postgres login and connecting, and another fetching MinIO credentials and doing a bucket create/write/read).

| Pool | Type | Backing device | Size | Used | Content |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `local` | dir | NVMe (pve-root) | 94 GiB | ~6.5% | iso, vztmpl (Debian 13 LXC template), backup, import |
| `local-lvm` | lvmthin (`pve/data`) | NVMe `nvme0n1` (Lexar NM790, 954 GB) | 816 GiB | ~1.4% | rootdir, images — template 9000 (8 GiB), `edge` rootfs (4 GiB), `vault` rootfs (8 GiB), `k3s-1` disk (60 GiB) + cloudinit, `pg` rootfs (8 GiB), `minio` rootfs (8 GiB) |
| `ssd-data` | lvmthin (`ssd-data/data`) | SATA SSD `sda` (238 GB) | 233.5 GiB nominal (thin-provisioned) | ~1.5% physical | rootdir, images — `vault`'s Raft data (5 GiB, `/vault/data`), `pg`'s data (40 GiB, `/pg/data`), `minio`'s data (120 GiB, `/minio/data`). ~165 GiB nominal claimed of the 233.5 GiB pool; physical usage still low since Postgres/MinIO hold little real data yet |

Storage tiering is deliberate: OS/VM/CT disks on fast NVMe, stateful data isolated on the SSD pool (`change-log/0002`). `vault`, `pg`, and `minio` each use a second, separate mount point (`mp0`) alongside their rootfs — the `terraform/modules/lxc` module's optional `data_disk` block (added in 0005) is now used by all three. `k3s-1` is deliberately **stateless/cluster-managed only** on `local-lvm` — no `ssd-data` allocation — consistent with 0006/0007's cloud-native separation (stateful backing services live in `pg`/`minio`, not in the compute tier). Because `ssd-data` is thin-provisioned, nominal claims (~165 GiB) already exceed a third of physical capacity while actual writes remain small — worth watching as Postgres/MinIO accumulate real data (no alerting on this yet; planned for `change-plan/0008`).

## Identities & access
| Identity | Realm | Role / scope | Purpose |
| :--- | :--- | :--- | :--- |
| `root@pam` | pam | superuser | host administration (SSH) |
| `terraform@pve` | pve | `TerraformProv` (22 privileges) at `/`, propagate | least-privilege IaC identity for Terraform — unchanged in privilege scope by 0004/0005 (no new PVE privileges were needed; the credential itself now lives in Vault, not a gitignored file) |

- **API token**: `terraform@pve!provider` — the credential Terraform authenticates with, **now sourced from Vault** (`kv/proxmox/terraform`) instead of a gitignored tfvars file.
- **In-guest (VMs)**: bootstrap user `debian` (cloud-init, key-only) → Ansible creates `deploy`. Unlike LXCs, `debian` is never locked out (the `baseline` role's SSH hardening only disables `root`), so it remains usable for every future re-run — confirmed with `k3s-1` in 0006, which never hit the LXC root-lockout gotcha.
- **In-guest (LXCs)**: bootstrap as `root` (Proxmox-injected key) → Ansible creates `deploy`, then disables root/password SSH. This is a one-shot window — see the IaC table note above.
- **Bastion access**: private-subnet guests are reached only via a jump through `edge` (`deploy@192.168.0.10`) — no direct LAN route exists to `10.10.10.0/24`. **Exception**: `edge` DNATs LAN `tcp/80,443` directly to `k3s-1`'s ingress (0006) — everything else on the private subnet, including `pg`/`minio`, stays bastion/tunnel-only.
- **Secrets**: the Proxmox token and Terraform's Vault AppRole credentials are the only local secrets remaining, both gitignored (`terraform/secrets.auto.tfvars`); the Proxmox token's authoritative copy is now Vault's `kv/proxmox/terraform`. Postgres and MinIO credentials are Vault-mediated (dynamic lease / KV respectively — see the Managed-services tier section) and never touch this repo. Workstation-side Vault/k3s trust material (`vault-ca.crt`, `k3s-ca.crt`, kubeconfig, the k3s reviewer JWT) lives outside the repo under `~/.vault-files/` and `~/.kube/` (renamed from `~/.vault/` in 0006 — that path collides with the Vault CLI's own reserved config file). Secret values are never stored in this repo or this doc.
