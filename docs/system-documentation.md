## Overview
A single **Dell OptiPlex 7070** Proxmox node being built, in code, as a cloud-analog DevOps + Data-Engineering platform (see `change-log/0001`). The platform has a working **IaC control plane** (`change-log/0003`) and now a **private-subnet network** (`change-log/0004`): a no-uplink bridge with an LXC edge acting as NAT gateway, SSH bastion, and DHCP/DNS for the private side. One guest runs persistently (`edge`); the golden template remains stopped. Honest framing: single node → this demonstrates cloud-native *patterns* (VPC + bastion), not production HA or physical network isolation.

```
LAN 192.168.0.0/24  (gw 192.168.0.1)
        │
        └── Proxmox host  "pve"  192.168.0.113  ── bridge vmbr0 ── nic0 (1 GbE)
              │
              ├── local-lvm   (NVMe, 816 GiB, ~0.23% used)   VM/CT root disks
              │     ├── VMID 9000 "debian13-cloud"  — golden VM template, stopped
              │     └── VMID  101 "edge"             — LXC, running (4 GiB disk)
              ├── ssd-data    (SATA SSD, 233 GiB, 0% used)  stateful data volumes  [empty]
              │
              └── vmbr1  (10.10.10.0/24, no bridge-ports — no LAN uplink)
                    └── edge (LXC 101)  eth1 10.10.10.1  — NAT gateway + SSH bastion + DHCP/DNS
                          (private-subnet guests — vault/k3s/pg/minio — attach here in 0005–0007; none yet)
```
`edge` is dual-homed: `eth0` → `vmbr0` (`192.168.0.10`, LAN-reachable), `eth1` → `vmbr1` (`10.10.10.1`, private gateway). A private-subnet guest reaches the internet only through `edge`'s NAT and is reachable from the LAN only by SSH-jumping through it (`ssh -J deploy@192.168.0.10 ...`) — verified end-to-end with a throwaway DHCP client (`priv-test`, VMID 998, destroyed after verification).

## IaC control plane (`change-log/0003`, extended by 0004)
| Component | Detail |
| :--- | :--- |
| Provisioning | Terraform ≥1.9, provider `bpg/proxmox` (installed: v0.111.1), state local & gitignored (`terraform/*.tfstate`) |
| VM path | `terraform/modules/vm` clones golden template 9000; two-phase guest-agent flip (Ansible installs it, second apply enables the channel) |
| LXC path (new, 0004) | `terraform/modules/lxc` — builds an unprivileged container from a downloaded `vztmpl` (not cloned from a template, since LXC has no equivalent to a QEMU template); parametrized multi-NIC `network_interfaces` list, one `ip_config` per interface in declared order |
| Container template | Debian 13 `debian-13-standard_13.1-2_amd64.tar.zst`, downloaded via `proxmox_virtual_environment_download_file` (content type `vztmpl`, md5-checksummed) from `download.proxmox.com` |
| Network as code | `proxmox_network_linux_bridge.vmbr1` — the private bridge is Terraform-managed, not GUI-created |
| Configuration | Ansible ≥2.16; `baseline` role now guards `qemu-guest-agent` with `ansible_virtualization_type != 'lxc'` (the agent is meaningless inside a container — a real compatibility fix made during 0004, applicable to every future LXC guest); new `edge` role: IPv4 forwarding, nftables ruleset, dnsmasq |
| LXC bootstrap pattern | Proxmox writes the SSH key straight into the container's `root` account at creation (no cloud-init) → Ansible connects as `root` once, creates `deploy`, disables root/password login — same bootstrap-then-harden shape as the VM path's `debian` user |

## Identities & access
| Identity | Realm | Role / scope | Purpose |
| :--- | :--- | :--- | :--- |
| `root@pam` | pam | superuser | host administration (SSH) |
| `terraform@pve` | pve | `TerraformProv` (22 privileges) at `/`, propagate | least-privilege IaC identity for Terraform — unchanged by 0004 (Proxmox's `VM.*` privilege namespace already covers LXC containers, so no new grants were needed to add the container/bridge resources) |

- **API token**: `terraform@pve!provider` (privsep 0) — the credential Terraform authenticates with.
- **In-guest (VMs)**: bootstrap user `debian` (cloud-init, key-only) → Ansible creates `deploy`.
- **In-guest (LXCs)**: bootstrap as `root` (Proxmox-injected key) → Ansible creates `deploy`, then disables root/password SSH.
- **Bastion access**: private-subnet guests are reached only via `ssh -J deploy@192.168.0.10 <user>@<private-ip>` — no direct LAN route exists to `10.10.10.0/24`.
- **Secrets**: the Proxmox token lives in a gitignored `terraform/secrets.auto.tfvars`; it will migrate to **Vault** (`change-plan/0005`). No Vault yet. Secret values are never stored in this repo or this doc.
