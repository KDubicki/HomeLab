# System Documentation (As-Built)

> **As-of:** 2026-07-06 · **Proxmox VE 9.2.4** (kernel 7.0.14-2-pve) · generated by the `architect-docs` skill from a live `ssh proxmox` scan. This is a **reference of reality** (what exists now); for *why*, see `change-plan/`/`change-log/`; for *how* it was built, see `runbooks/` (deleted once Executed) and the repo's `terraform/`/`ansible/` trees.

## Overview
A single **Dell OptiPlex 7070** Proxmox node being built, in code, as a cloud-analog DevOps + Data-Engineering platform (see `change-log/0001`). The platform has a working **IaC control plane** (`change-log/0003`), a **private-subnet network** (`change-log/0004`), a **secrets manager** on the private subnet (single-node HashiCorp Vault — `change-log/0005`), and now a **compute tier**: single-node k3s with a live HashiCorp Vault Agent Injector (`change-log/0006`). Three guests run persistently (`edge`, `vault`, `k3s-1`); the golden template remains stopped. Honest framing: single node → this demonstrates cloud-native *patterns* (VPC + bastion, centralized secrets with least-privilege auth, managed-Kubernetes-style compute), not production HA, physical network isolation, or a highly-available Vault/k8s cluster.

```
LAN 192.168.0.0/24  (gw 192.168.0.1)
        │
        └── Proxmox host  "pve"  192.168.0.113  ── bridge vmbr0 ── nic0 (1 GbE)
              │
              ├── local-lvm   (NVMe, 816 GiB, ~1.0% used)   VM/CT root disks
              │     ├── VMID 9000 "debian13-cloud"  — golden VM template, stopped
              │     ├── VMID  101 "edge"             — LXC, running (4 GiB disk)
              │     ├── VMID  102 "vault"            — LXC, running (8 GiB disk)
              │     └── VMID  110 "k3s-1"            — VM, running (60 GiB disk)
              ├── ssd-data    (SATA SSD, 233.5 GiB, ~0.06% used)  stateful data volumes
              │     └── vm-102-disk-0 (5 GiB)  — Vault's Raft data mount (/vault/data)
              │
              └── vmbr1  (10.10.10.0/24, no bridge-ports — no LAN uplink)
                    ├── edge (LXC 101)   eth1 10.10.10.1   — NAT gateway + SSH bastion + DHCP/DNS + DNAT (80/443 → k3s-1)
                    ├── vault (LXC 102)  eth0 10.10.10.10  — HashiCorp Vault, TLS :8200, private-subnet only (no LAN NIC)
                    └── k3s-1 (VM 110)   eth0 10.10.10.20  — single-node k3s v1.36.2+k3s1, private-subnet only
                          (pg/minio attach here in 0007; none yet)
```
`edge` is dual-homed: `eth0` → `vmbr0` (`192.168.0.10`, LAN-reachable), `eth1` → `vmbr1` (`10.10.10.1`, private gateway). `vault` and `k3s-1` each have a single NIC on `vmbr1` only — reachable **only** by SSH-jumping through `edge`, or via an SSH local-port-forward tunnel for their respective APIs (Vault: `-L 8200:10.10.10.10:8200`; k3s kube-API: `-L 6443:10.10.10.20:6443`). `edge` additionally DNATs LAN-side `tcp/80,443` straight to `k3s-1` (`change-log/0006`) — the one deliberate exception to "bastion-only": ingress traffic is meant to be reached directly, the way a cloud ALB fronts a private-subnet target, while the cluster's admin plane (kube-API) stays tunnel-only like Vault's. These patterns were proven end-to-end during 0004 (throwaway DHCP client `priv-test`, destroyed after verification), 0005 (Vault init/unseal, secrets-engine setup, a live `terraform plan` reading a secret over the tunnel), and 0006 (a sample Kubernetes Deployment/Ingress reachable via the DNAT, with a Vault-injected secret, and `kubectl` reachable via the tunnel).

Storage tiering is deliberate: OS/VM/CT disks on fast NVMe, stateful data isolated on the SSD pool (`change-log/0002`). `vault` is the only guest using a second, separate mount point (`mp0`) alongside its rootfs — the `terraform/modules/lxc` module gained an optional `data_disk` block in 0005 to support this. `k3s-1` is deliberately **stateless/cluster-managed only** on `local-lvm` — no `ssd-data` allocation — consistent with 0006's cloud-native separation (stateful backing services live in 0007, not in the compute tier).

## IaC control plane (`change-log/0003`, extended by 0004, 0005, and 0006)
| Component | Detail |
| :--- | :--- |
| Provisioning | Terraform ≥1.9, provider `bpg/proxmox` (installed: v0.111.1) + `hashicorp/vault` (installed: v5.10.1, added in 0005), state local & gitignored (`terraform/*.tfstate`) |
| VM path | `terraform/modules/vm` clones golden template 9000 (used by `k3s-1` in 0006, alongside earlier throwaway tests); two-phase guest-agent flip — Terraform attaches the virtio-serial channel (`agent_enabled=true`) **before** Ansible installs/starts the service, not after (corrected step order, 0006). On the current provider (`bpg/proxmox` 0.111.1) that flip performs a `qmshutdown`+`qmstart`, not a true hot-plug as originally assumed in 0003 — harmless pre-workload, but not zero-downtime. The module also hardcodes guest DNS to `192.168.0.1` regardless of bridge, so a private-subnet VM cannot resolve `edge`'s dnsmasq zone (found in 0006, worked around with IP addressing, carried as a follow-up) |
| LXC path | `terraform/modules/lxc` — builds an unprivileged container from a downloaded `vztmpl`; parametrized multi-NIC `network_interfaces`; an optional `data_disk` block (added in 0005) adds a second `mount_point` volume on a separate datastore (used by `vault`'s Raft storage) |
| Container template | Debian 13 `debian-13-standard_13.1-2_amd64.tar.zst`, downloaded via `proxmox_virtual_environment_download_file` (content type `vztmpl`, md5-checksummed) from `download.proxmox.com` |
| Network as code | `proxmox_network_linux_bridge.vmbr1` — the private bridge is Terraform-managed, not GUI-created |
| Secrets as code | `terraform/secrets.tf` — a `vault` provider (AppRole login, `skip_child_token = true` to stay within the least-privilege policy) plus a `vault_kv_secret_v2` data source feed the `proxmox` provider's API token (`change-log/0005`) |
| Configuration | Ansible ≥2.16; `baseline` role guards `qemu-guest-agent` with `ansible_virtualization_type != 'lxc'`; `edge` role: IPv4 forwarding, nftables ruleset, dnsmasq (now with a static host record for `vault.lab.internal`); new `vault` role (0005): installs Vault 1.17.6, generates a self-signed TLS cert, writes the Raft/TLS config and systemd unit |
| LXC bootstrap pattern | Proxmox writes the SSH key straight into the container's `root` account at creation (no cloud-init) → Ansible connects as `root` once, creates `deploy`, disables root/password login. **Operational note from 0005**: this means `root` SSH access is a *one-time* window — every later connection (Ansible re-runs, manual admin) must use `deploy` + `sudo`, and this OpenSSH client build (10.2p1) does not propagate `-i` across a `ProxyJump` hop with no agent key loaded, so jumps through `edge` use an explicit `ProxyCommand` instead |

## Identities & access
| Identity | Realm | Role / scope | Purpose |
| :--- | :--- | :--- | :--- |
| `root@pam` | pam | superuser | host administration (SSH) |
| `terraform@pve` | pve | `TerraformProv` (22 privileges) at `/`, propagate | least-privilege IaC identity for Terraform — unchanged in privilege scope by 0004/0005 (no new PVE privileges were needed; the credential itself now lives in Vault, not a gitignored file) |

- **API token**: `terraform@pve!provider` — the credential Terraform authenticates with, **now sourced from Vault** (`kv/proxmox/terraform`) instead of a gitignored tfvars file.
- **In-guest (VMs)**: bootstrap user `debian` (cloud-init, key-only) → Ansible creates `deploy`. Unlike LXCs, `debian` is never locked out (the `baseline` role's SSH hardening only disables `root`), so it remains usable for every future re-run — confirmed with `k3s-1` in 0006, which never hit the LXC root-lockout gotcha.
- **In-guest (LXCs)**: bootstrap as `root` (Proxmox-injected key) → Ansible creates `deploy`, then disables root/password SSH. This is a one-shot window — see the IaC table note above.
- **Bastion access**: private-subnet guests are reached only via a jump through `edge` (`deploy@192.168.0.10`) — no direct LAN route exists to `10.10.10.0/24`. **Exception**: `edge` DNATs LAN `tcp/80,443` directly to `k3s-1`'s ingress (0006) — everything else on the private subnet stays bastion/tunnel-only.
- **Secrets**: the Proxmox token and Terraform's Vault AppRole credentials are the only local secrets remaining, both gitignored (`terraform/secrets.auto.tfvars`); the Proxmox token's authoritative copy is now Vault's `kv/proxmox/terraform`. Workstation-side Vault/k3s trust material (`vault-ca.crt`, `k3s-ca.crt`, kubeconfig, the k3s reviewer JWT) lives outside the repo under `~/.vault-files/` and `~/.kube/` (renamed from `~/.vault/` in 0006 — that path collides with the Vault CLI's own reserved config file). Secret values are never stored in this repo or this doc.
